# AI Life - 时间线设计

## 核心原则

1. **Timeline 是产品，不是 AI 分析**：用户打开时间线，看到的应该是"我的生活"，而不是"AI 对我的分析"
2. **只呈现，不评判**：不要判断重要不重要、好不好、对不对
3. **不做业务分类**：不要给事件打上"工作""生活""健康"等分类标签
4. **用用户的语言**：标题和摘要要自然、口语化，像用户自己在记录一样
5. **Fact 提供结构，Message 提供细节**：从 Facts 中提取结构化信息（时间、人物、行为），但语言风格和细节从原始消息中获取

## 输出字段

### title（必填）

**一句话标题，简洁有力，像日记标题一样。**

**要求**：
- 不超过 30 个字
- 要包含核心事件
- 自然、口语化

**示例**：
- ✅ "女儿第一次学会骑车"
- ✅ "今天开始创业了"
- ✅ "和老朋友吃了顿火锅"
- ❌ "用户在公园陪女儿骑自行车"（太技术化）

### summary（必填）

**两三句话的摘要，补充细节和感受。**

**要求**：
- 2-3 句话
- 可以包含情绪和氛围
- 比 title 更丰富，但不要太冗长

**示例**：
- "今天陪女儿去公园练习骑车，练了一个多小时，她终于第一次自己骑起来了。你特别开心，觉得孩子长大了。"

### occurred_at（必填）

**事件发生的时间，ISO 8601 格式。**

**规则**：
1. 如果有明确日期，用具体日期和时间
2. 如果是相对时间（今天/昨天/上周），需要根据消息时间推算
3. 如果时间不明确，**使用消息创建时间**（包含真实时分秒）
4. **不要使用 00:00:00**，除非用户明确说是凌晨零点

**示例**：
- "2024-07-22T14:30:00.000Z"
- "2024-07-21T09:15:00.000Z"

### time_precision（必填）

**时间精度，只能是以下值之一**：

| 值 | 判断标准 |
|----|---------|
| day | 用户说"今天/昨天/7月21日" |
| week | 用户说"上周/这周" |
| month | 用户说"上个月/今年3月" |
| year | 用户说"去年/2023年" |
| unknown | 用户说"小时候/以前/曾经" |

### icon（必填）

**一个 emoji 图标，代表这个事件。**

**常见类型参考**：

| 类别 | emoji | 适用场景 |
|------|-------|---------|
| 孩子/家庭 | 👶 | 女儿、儿子、家人相关 |
| 工作/事业 | 💼 | 工作、创业、项目 |
| 学习/读书 | 📚 | 学习、读书、考试 |
| 运动/健康 | 🏃 | 跑步、健身、身体状态 |
| 美食/饮食 | 🍜 | 吃饭、美食、聚餐 |
| 旅行/出行 | ✈️ | 旅行、出差、出门 |
| 庆祝/开心 | 🎉 | 生日、庆祝、喜事 |
| 难过/失落 | 😢 | 悲伤、失落、难过 |
| 想法/灵感 | 💡 | 想法、灵感、创意 |
| 社交/朋友 | 🤝 | 朋友、社交、聚会 |
| 生活/日常 | 🏠 | 日常生活、家务 |
| 不确定 | ❓ | 无法确定类别时 |

## 输入数据

Timeline Agent 接收以下输入：

1. **originalMessage**：用户原始消息文本
2. **factGroup**：事实组（包含 summary 和 fact_count）
3. **facts**：事实列表（9种类型的结构化事实）
4. **messageCreatedAt**：消息发送时间（ISO 8601）

## 输入输出示例

### 示例 1

**输入**：
- originalMessage: "今天陪女儿去公园骑自行车，练了一个多小时，她终于第一次自己骑起来了，特别开心"
- messageCreatedAt: "2024-07-22T14:30:00.000Z"
- facts: [...包含 action=骑车, person=女儿, location=公园, time=今天, emotion=开心...]

**输出**：
```json
{
  "title": "女儿第一次学会骑车",

}
```

## 数据模型

### timeline 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| user_id | UUID | 用户 ID |
| message_id | UUID | 关联消息 ID |
| fact_group_id | UUID | 关联事实组 ID |
| title | TEXT | 事件标题（≤30字） |
| summary | TEXT | 事件摘要（2-3句话） |
| occurred_at | TIMESTAMP | 事件发生时间 |
| time_precision | TEXT | 时间精度（day/week/month/year/unknown） |
| icon | TEXT | emoji 图标 |
| event_source | TEXT | 事件来源（chat/photo/voice/calendar/document/health） |
| raw_content | TEXT | AI 原始响应 |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |

### event_source 枚举值

| 值 | 说明 |
|----|------|
| chat | 聊天消息 |
| photo | 照片上传 |
| voice | 语音消息 |
| calendar | 日历集成 |
| document | 文档上传 |
| health | 健康数据 |

## 处理流程

### 实时生成流程

```
用户发送消息 → 事实抽取完成 → Timeline Agent → 保存到 timeline 表
                                                   ↓
                                           更新操作日志
```

### 批量生成流程

```
获取所有 fact_groups → 筛选未生成时间线的消息 → 逐条调用 Timeline Agent → 保存到 timeline 表
```

## API 接口

### POST /api/generate-timeline/all

**功能**：批量为所有未生成时间线的消息生成时间线事件

**请求体**：无

**响应**：
```json
{
  "success": true,
  "count": 86,
  "success_count": 55,
  "failed_count": 0,
  "failed_messages": []
}
```

## UI 设计

### 时间线页面

**布局**：
- 左侧：时间轴（宽度 80px），显示日期和时间
- 右侧：事件卡片列表

**时间轴样式**：
- 垂直连线
- 日期标签（年/月/日）
- 时间标签（时:分）
- 圆形节点标记事件位置

**事件卡片**：
- emoji 图标（左侧）
- 标题（加粗）
- 摘要（灰色小字）
- 时间精度标签（右上角）

**缩放功能**：
- 按天：显示每一天的事件
- 按周：按周分组显示
- 按月：按月分组显示
- 按年：按年分组显示

**筛选功能**：
- 按事件来源筛选（chat/photo/voice等）
- 按关键词搜索

**详情弹窗**：
- 完整摘要
- 关联消息链接
- 关联事实列表
- 关闭按钮（右上角）
- 支持拖拽关闭

## 关键设计决策

1. **不做业务分类**：时间线是产品，展示"发生了什么"，分类交给 Memory 层
2. **用户语言优先**：标题和摘要使用用户口语化表达，避免技术术语
3. **真实时间记录**：不使用 00:00:00，确保时间精准性
4. **时间精度标注**：每个事件标注精度，帮助用户理解时间确定性
5. **emoji 图标**：用视觉符号快速识别事件类型，提升可读性
6. **分离事实与时间线**：事实是数据层，时间线是产品层，支持重新生成
7. **批量生成支持**：允许从已有事实重新生成时间线，便于迭代优化
